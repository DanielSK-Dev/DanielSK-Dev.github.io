Churp — Peer-to-Peer Messenger Framework

Capstone project (2025) · Role: Built all code; partner helped with theory/requirements.

Links

Source
Repo

1) Problem

I wanted a reusable peer-to-peer messaging framework that could send structured data between two clients without requiring deep networking knowledge every time I reused it. The biggest blocker for real-world usability was NAT/firewall friction — many users can’t easily port-forward.

2) Goals & constraints

  • Reusable framework (not just a one-off app)
  • Structured payloads (JSON messages, consistent schema)
  • Practical connectivity through NAT traversal techniques
  • Debuggable: logs, clear state transitions, predictable failure modes
  • Constraints: consumer NATs, different networks, limited time (capstone timeline)

3) Solution overview

Churp is a framework that establishes a peer connection and sends messages as JSON objects. It includes connection setup steps, message serialization/deserialization, and a clean API for adding message types. For NAT traversal, it uses UDP hole punching (with a coordination step) so peers can connect in more common home network setups without manual port forwarding in many cases.

4) Key technical highlights

  • Protocol design: defined message envelopes and versioning to keep payloads consistent.
  • State machine: explicit connection phases (init → rendezvous → punch → active → teardown).
  • Input validation: reject malformed JSON and unexpected message types safely.
  • Logging strategy: actionable logs for connectivity and parsing failures.

5) Hard problems & how I solved them

  • Challenge: unreliable connection attempts across NATs → Investigation: analyzed packet flow + timeouts → Fix: retries/backoff + clearer phases → Result: more consistent connect behavior.
  • Challenge: ambiguous failures when peers disagree on message format → Fix: strict schema checks + helpful error responses.

6) Results

  • Working P2P framework with JSON messaging and a clean API for message types.

7) Security considerations

  • Abuse resistance: validate payload size/type; reject unknown message types.
  • Trust boundaries: treat all inbound messages as untrusted.

8) What I’d do next

  • Optional encryption (e.g., key exchange + secure channel)
  • Better NAT traversal coverage (TURN/relay fallback)
  • Packaging + CLI tools for quick usage
